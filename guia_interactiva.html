🎮 1. Visión General del Proyecto

¿Qué es este proyecto? ▼

Este proyecto es un sistema de reseñas de videojuegos compuesto por DOS aplicaciones separadas que se comunican entre sí mediante una API REST.

Aplicación Principal (PHP): Permite registrar videojuegos, escribir reseñas con calificaciones de 1 a 5 estrellas, consultar el catálogo completo y ver estadísticas generadas por el servicio de analítica.

Servicio de Analítica (Flask): Recibe datos de reseñas desde PHP, los almacena en MongoDB, y genera estadísticas (totales, promedios, rankings de mejores videojuegos).

Ambos sistemas están desplegados en Render (plataforma cloud gratuita) usando Docker. PostgreSQL está en Render y MongoDB en MongoDB Atlas.

URLs públicas:

🌐 PHP: https://resenas-php.onrender.com
🐍 Flask: https://resenas-flask.onrender.com

¿Por qué dos bases de datos diferentes?

PostgreSQL (SQL) para la app principal: datos estructurados, relaciones entre tablas (1 videojuego → muchas reseñas), integridad referencial con FOREIGN KEY y ON DELETE CASCADE, consultas JOIN eficientes.

MongoDB (NoSQL) para analítica: datos agregados (sumas, promedios, conteos), sin esquema fijo, más rápido para escrituras frecuentes, pipelines de agregación potentes.

🗄️ 3. Base de Datos PostgreSQL

Tabla: videojuegos ▼
CREATE TABLE videojuegos (
  id SERIAL PRIMARY KEY,
  nombre VARCHAR(200) NOT NULL,
  genero VARCHAR(100) NOT NULL,
  plataforma VARCHAR(100) NOT NULL,
  descripcion TEXT,
  fecha_lanzamiento DATE,
  fecha_registro TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

SERIAL: Genera números autoincrementales (1, 2, 3...). PRIMARY KEY: Identificador único, no puede repetirse. VARCHAR(200): Texto variable de máximo 200 caracteres, más eficiente que TEXT para strings cortos. TEXT: Texto ilimitado (hasta 1GB), ideal para descripciones largas. DATE: Solo fecha (año-mes-día), ocupa 4 bytes. TIMESTAMP: Fecha + hora, ocupa 8 bytes. DEFAULT CURRENT_TIMESTAMP: Si no se especifica valor, usa la fecha/hora actual automáticamente.

Tabla: resenas (con restricciones) ▼
CREATE TABLE resenas (
  id SERIAL PRIMARY KEY,
  videojuego_id INTEGER NOT NULL,
  nombre_usuario VARCHAR(150) NOT NULL,
  calificacion INTEGER NOT NULL CHECK (calificacion >= 1 AND calificacion <= 5),
  comentario TEXT,
  fecha_registro TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_resena_videojuego
    FOREIGN KEY (videojuego_id) REFERENCES videojuegos(id)
    ON DELETE CASCADE
);

CHECK (calificacion >= 1 AND calificacion <= 5): Restricción que impide guardar valores fuera del rango 1-5. Si alguien intenta calificacion=6, PostgreSQL rechaza la operación.

FOREIGN KEY: Garantiza que videojuego_id siempre apunte a un videojuego que existe. No se puede insertar una reseña para un juego inexistente.

ON DELETE CASCADE: Si se borra un videojuego, se borran TODAS sus reseñas automáticamente. Evita registros "huérfanos".

🍃 4. Base de Datos MongoDB Atlas

Estructura del documento y operaciones ▼

Base de datos: analitica_videojuegos
Colección: estadisticas_videojuegos

{
  "videojuego_id": 1,
  "nombre_videojuego": "The Legend of Zelda: Tears of the Kingdom",
  "total_resenas": 5,
  "suma_calificaciones": 22,
  "promedio_calificacion": 4.4,
  "ultima_resena": "2026-07-01T12:00:00Z"
}

¿Por qué guardar suma y contador en vez de cada reseña? Es el patrón "aggregation by accumulation". En vez de guardar cada reseña individual y recalcular todo desde cero, mantenemos acumulados. Ventajas: consultas más rápidas (leer 1 doc vs N docs), menos espacio, el promedio es una simple división.

Pipeline de agregación usado en /api/estadisticas:

[
  { '$group': {
    '_id': None,
    'total_resenas': {'$sum': '$total_resenas'},
    'suma_calificaciones': {'$sum': '$suma_calificaciones'}
  }
]

Agrupa todos los documentos en un solo grupo y suma los totales de reseñas y calificaciones de toda la colección.

🐘 5. Aplicación PHP - Archivo por Archivo

conexion.php — Conexión PDO a PostgreSQL ▼

Establece la conexión con PostgreSQL usando PDO (PHP Data Objects). Lee variables de entorno (getenv()) para host, puerto, nombre BD, usuario y contraseña. Si no existen (desarrollo local), usa valores por defecto.

DSN: "pgsql:host=$host_bd;port=$puerto_bd;dbname=$nombre_bd"

Configuraciones clave:

  • PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION: Lanza excepciones cuando algo falla, permitiendo usar try/catch.
  • PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC: Los resultados se devuelven como arrays asociativos ($fila['nombre']).
  • PDO::ATTR_EMULATE_PREPARES => false: Usa preparación nativa de PostgreSQL (más segura).

¿Por qué variables de entorno? Para no escribir credenciales en el código. En desarrollo se usan defaults, en producción Render inyecta los valores reales.

registrar_videojuego.php — Formulario con validaciones ▼

Flujo: GET muestra el formulario vacío, POST valida los campos e inserta en la BD.

Validaciones por campo:

  • nombre: Obligatorio (empty()), mínimo 2 caracteres, máximo 200 (strlen())
  • genero: Obligatorio, select con opciones predefinidas
  • plataforma: Obligatorio, select con opciones predefinidas
  • descripción: Opcional, máximo 2000 caracteres
  • fecha_lanzamiento: Obligatorio, se valida con checkdate() que verifica fechas reales

Seguridad: htmlspecialchars() previene XSS, consultas preparadas con bindParam() previenen inyección SQL, validación del lado del servidor (no se confía en HTML5).

registrar_resena.php — Integración PHP → Flask ▼

El select de videojuegos se llena dinámicamente desde PostgreSQL. Al registrar, además de guardar en PostgreSQL, envía datos a la API Flask mediante cURL.

Flujo de integración: Formulario → Validación → INSERT en PostgreSQL → cURL POST a Flask → Flask actualiza MongoDB.

Se usa curl_init(), curl_setopt_array(), curl_exec() y curl_close() para enviar JSON con los datos de la reseña a la API Flask.

catalogo.php — Vista pública con JOIN y estrellas ▼
SELECT v.*,
  COALESCE(ROUND(AVG(r.calificacion)::numeric, 1), 0) AS promedio,
  COUNT(r.id) AS total_resenas
FROM videojuegos v
LEFT JOIN resenas r ON v.id = r.videojuego_id
GROUP BY v.id ORDER BY promedio DESC

LEFT JOIN: Incluye videojuegos sin reseñas (promedio = 0). AVG(): Promedio de calificaciones. ROUND(..., 1): Redondea a 1 decimal. COALESCE(..., 0): Si es NULL, usa 0.

Las estrellas se muestran con caracteres Unicode: ★ (U+2605, llena) y ☆ (U+2606, vacía), en un bucle for que redondea el promedio.

funciones_api.php — Comunicación con Flask ▼

Contiene 3 funciones:

enviar_resena_a_api(): POST a /api/videojuegos usando cURL. Envía JSON con videojuego_id, calificacion, nombre_videojuego.

consultar_estadisticas_api(): GET a /api/estadisticas usando file_get_contents() con stream context.

consultar_mejores_videojuegos_api(): GET a /api/mejores-videojuegos usando file_get_contents().

¿Por qué cURL para POST y file_get_contents para GET? cURL es más potente para POST con JSON y headers personalizados. Para GET simples, file_get_contents es suficiente y más simple.

🐍 6. API Flask - Endpoints

app.py — Los 4 endpoints ▼
MétodoEndpointPropósito
GET/Verificar que la API funciona
POST/api/videojuegosRecibir reseña, actualizar estadísticas en MongoDB
GET/api/estadisticasDevolver totales, promedios, mejor juego, más reseñado
GET/api/mejores-videojuegosRanking ordenado por promedio de calificación

POST /api/videojuegos — Lógica:

  1. Recibe JSON con videojuego_id, calificacion, nombre_videojuego
  2. Valida campos obligatorios y calificación entre 1 y 5
  3. Busca el videojuego en MongoDB con find_one()
  4. Si EXISTE: incrementa total_resenas, suma_calificaciones, recalcula promedio, usa update_one()
  5. Si NO EXISTE: crea documento nuevo con insert_one()
  6. Responde 201 Created con mensaje de confirmación

En producción Render ejecuta: gunicorn app:aplicacion --bind 0.0.0.0:$PORT

🔄 7. Flujo Completo de una Reseña

Los 8 pasos de una reseña (click para expandir) ▼
1

Usuario llena el formulario

Selecciona un videojuego del menú (cargado desde PostgreSQL), escribe su nombre, elige calificación 1-5, escribe comentario opcional. Clic en "Registrar Reseña".

2

PHP recibe los datos (POST)

El formulario se envía por POST a registrar_resena.php. Los datos llegan en el array superglobal $_POST.

3

Validación de campos

Verifica videojuego_id, existencia del juego en BD, longitud del nombre, rango de calificación, longitud del comentario. Si falla, muestra errores en rojo.

4

Inserción en PostgreSQL

Ejecuta INSERT con consultas preparadas y bindParam(). Los marcadores :videojuego_id, :nombre_usuario se reemplazan de forma segura.

5

Notificación a la API Flask (cURL)

PHP prepara un array, lo convierte a JSON con json_encode(), configura cURL para POST a /api/videojuegos con header Content-Type: application/json.

6

Flask actualiza MongoDB

El endpoint recibe el JSON, busca el documento. Si existe: incrementa contador y recalcula promedio. Si no: crea documento nuevo. Responde 201.

7

PHP muestra resultado al usuario

Mensaje verde: "¡Reseña registrada correctamente! Se enviaron datos a la API de analítica."

8

Consulta de estadísticas

Al visitar estadisticas.php, PHP hace GET a /api/estadisticas, Flask consulta MongoDB con aggregate() y devuelve los datos calculados.

🐳 8. Docker y Despliegue en Render

¿Qué es Docker y cómo se usa en el proyecto? ▼

Docker es una plataforma que "empaqueta" una aplicación con todas sus dependencias en un contenedor. Un contenedor es como una máquina virtual ligera que incluye SO, bibliotecas, y la aplicación.

Ventajas: "En mi máquina funciona" deja de ser problema. El mismo contenedor funciona en desarrollo y producción. Fácil de desplegar (solo necesitas el Dockerfile). Aislamiento: cada servicio tiene sus propias dependencias.

Dockerfile de PHP: Basado en php:8.2-apache, instala libpq-dev y compila pdo_pgsql, copia los archivos a /var/www/html/.

Dockerfile de Flask: Basado en python:3.11-slim, instala dependencias con pip, ejecuta con gunicorn app:aplicacion --bind 0.0.0.0:$PORT.

docker-compose.yml: Solo para desarrollo local. Define 3 servicios: postgres, php (puerto 3000), flask (puerto 5000). Comando: docker compose up -d

Render: Al conectar el repo de GitHub, Render detecta los Dockerfiles y construye las imágenes automáticamente. Los servicios gratuitos se "duermen" tras 15 min de inactividad (la primera petición tarda 30-60s).

❓ 12. Preguntas Frecuentes del Profesor

Haz clic en cada pregunta para ver la respuesta preparada. Son 12 preguntas clave que cubren todo el proyecto.

¿Por qué usaste dos bases de datos diferentes? ▼

Porque cada una es mejor para su propósito. PostgreSQL es relacional, ideal para datos estructurados con relaciones (un videojuego → muchas reseñas). MongoDB es NoSQL, ideal para analítica con datos que se actualizan frecuentemente y no necesitan esquema rígido. Además, separar app principal de analítica permite que si una BD falla, la otra siga funcionando.

¿Cómo se comunican PHP y Flask? ▼

Mediante una API REST sobre HTTP. PHP actúa como cliente HTTP y envía peticiones a los endpoints de Flask usando cURL. Los datos se intercambian en formato JSON. Esta comunicación es independiente del lenguaje: PHP no sabe que Flask está en Python, solo conoce las URLs y el formato de los datos.

¿Qué es PDO y por qué lo usaste? ▼

PDO (PHP Data Objects) es una capa de abstracción de base de datos. Permite usar diferentes motores con la misma sintaxis. Su principal ventaja de seguridad son las consultas preparadas (prepare + bindParam) que evitan inyección SQL al separar la estructura SQL de los datos del usuario.

¿Cómo evitaste la inyección SQL? ▼

Usando consultas preparadas de PDO. Los valores del usuario nunca se concatenan en la consulta SQL. Se usan marcadores (:nombre, :genero) que PDO reemplaza de forma segura. El motor de BD recibe estructura y datos por separado, haciendo imposible modificar la estructura de la consulta.

¿Cómo implementaste las validaciones? ▼

Del lado del servidor en PHP. Por cada campo: empty() para obligatoriedad, strlen() para longitudes, is_numeric() para números, checkdate() para fechas válidas. Los errores se almacenan en un array $errores y se muestran junto a cada campo con clases Bootstrap is-invalid.

¿Por qué desplegaste en Render? ▼

Render ofrece plan gratuito con PostgreSQL administrado, soporte Docker, despliegue automático desde GitHub, certificados SSL, y variables de entorno. Heroku eliminó su plan gratuito, Railway tiene créditos limitados. Render permite ambos servicios (PHP y Flask) en el mismo ecosistema.

¿Qué son las variables de entorno? ▼

Valores de configuración definidos fuera del código, en el sistema operativo o plataforma cloud. Se usan para datos sensibles (contraseñas, URLs) que no deben ir en el código. En desarrollo local se usan defaults, en producción Render inyecta los reales. Así el mismo código funciona en cualquier entorno.

¿Cómo funciona el Dockerfile? ▼

Un Dockerfile es un archivo de instrucciones para construir una imagen Docker. Cada instrucción (FROM, RUN, COPY, CMD) crea una "capa". Las capas se cachean: si una instrucción no cambia, Docker reusa la capa anterior. La última instrucción (CMD) define el comando que se ejecuta al iniciar el contenedor.

¿Cómo se relacionan las tablas en PostgreSQL? ▼

La tabla resenas tiene una FOREIGN KEY que referencia videojuegos. Esto crea una relación "uno a muchos": un videojuego → muchas reseñas. ON DELETE CASCADE asegura que al borrar un juego, sus reseñas también se borren, manteniendo integridad referencial.

¿Qué hace POST /api/videojuegos? ▼

Recibe datos de una reseña (videojuego_id, calificacion, nombre) y actualiza estadísticas en MongoDB. Si es la primera reseña: crea documento con total_resenas=1. Si ya existe: incrementa contador, suma la calificación, recalcula promedio = suma_calificaciones / total_resenas. Responde 201 Created.

¿Cómo documentaste las pruebas? ▼

Selenium: comentarios en el propio archivo .py explicando cada paso. Locust: reporte HTML con gráficos y tablas de tiempos, throughput y errores. Postman: collection con test scripts en JavaScript que validan códigos HTTP, estructura JSON y contenido de respuestas.

¿Qué aprendiste con este proyecto? ▼

Aprendí a diseñar una arquitectura de microservicios simple con dos aplicaciones independientes comunicadas vía API REST. Trabajé con bases de datos SQL y NoSQL simultáneamente, contenericé aplicaciones con Docker, desplegué en la nube con Render, y realicé pruebas automatizadas con Selenium, de carga con Locust, y de integración con Postman.
